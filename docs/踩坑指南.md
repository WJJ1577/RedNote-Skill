## 二、踩坑记录

### Bug #1: websectiga 解码错误

**症状**：所有 edith API 返回 400 或 461

**根因**：`websectiga` 解码函数索引方式错误

```python
# ❌ RedNote 错误写法
key = [target_decode_list[675 + i][2] for i in range(0, 128, 2)]

# ✅ RedCrack 正确写法（需要用 d[] 二次索引）
key = [d[target_decode_list[675 + i][2]] for i in range(0, 128, 2)]
```

**关键差异**：`target_decode_list[i][2]` 返回的是**索引值**，需要用 `d[索引值]` 获取实际 key。

**影响**：解码出乱码 `z~~{{~~{~`，CDN 拒绝携带损坏 cookie 的请求。

**教训**：逆向时必须逐行对照，`d[xxx]` 不是多余的。

---

### Bug #2: HTTP 库 TLS 指纹被 CDN 拦截

**症状**：httpx 请求被 TencentEdgeOne/Akamai CDN 拦截，返回 400

**根因**：CDN 通过 TLS 握手指纹（JA3/JA4）识别非浏览器客户端

| HTTP 库 | TLS 指纹 | CDN 结果 |
|---------|---------|---------|
| `aiohttp` | ✅ 通过 | 200 |
| `httpx` | ❌ 被识别 | 400 |
| `curl` | ❌ 被识别 | 400/461 |

**解决方案**：将 HTTP 客户端从 `httpx` 换成 `aiohttp`（与 RedCrack 一致）

**教训**：反爬项目中，HTTP 库的选择至关重要。`aiohttp` 的 TLS 栈与浏览器更接近。

---

### Bug #3: 缺少 acw_tc Cookie

**症状**：搜索 API 返回 200 但 items 为空

**根因**：`acw_tc` 是 TencentEdgeOne CDN 的反爬 token，必须从 `edith.xiaohongshu.com/` 首页获取

```python
# 初始化时必须先访问首页获取 acw_tc
resp = await client.get("https://edith.xiaohongshu.com/")
# acw_tc 会在 Set-Cookie 中返回
```

**教训**：CDN 级别的 cookie 和应用级 cookie 是分开的，需要分别获取。

---

### Bug #4: 浏览器特征头触发 WAF

**症状**：httpx 自动添加的 `sec-ch-ua*`、`sec-fetch-*` 头被 CDN 识别为非浏览器

**根因**：这些头是浏览器自动添加的，CDN 会验证它们与 TLS 指纹的一致性

**解决方案**：在请求签名中间件中移除这些头

```python
for hdr in ("accept-encoding", "connection", "sec-ch-ua",
            "sec-ch-ua-mobile", "sec-ch-ua-platform",
            "sec-fetch-dest", "sec-fetch-mode", "sec-fetch-site", "priority"):
    request.headers.pop(hdr, None)
```

**教训**：不是"像浏览器"就行，要"完全一致"。TLS 指纹和 HTTP 头必须匹配。

---

### Bug #5: 浏览器指纹字段不完整

**症状**：gid 请求返回 400

**根因**：RedNote 只发送 15 字段的最小指纹，RedCrack 发送 80+ 字段的完整指纹

| 字段 | RedNote | RedCrack |
|------|---------|----------|
| x1 (User-Agent) | ❌ 缺失 | ✅ 完整 |
| x7 (GPU 信息) | ❌ 缺失 | ✅ 随机 GPU |
| x9 (屏幕分辨率) | ❌ 缺失 | ✅ 随机分辨率 |
| x22 (浏览器指纹) | ❌ 缺失 | ✅ MD5 hash |
| ... | ... | ... |

**解决方案**：实现 `XhsFpGenerator.get_fingerprint()`，生成完整 80+ 字段指纹

**教训**：反爬系统的指纹验证是**完整性校验**，缺少字段 = 非浏览器。

---

### Bug #6: URL 路由错误

**症状**：API 请求返回 500

**根因**：相对 URL 路由逻辑错误，`/api/*` 被发到 `www.xiaohongshu.com` 而非 `edith.xiaohongshu.com`

```python
# ❌ 错误路由
if "/api/sns/web/v2/login" in url:
    base = EDITH_BASE
else:
    base = API_BASE  # 所有其他 API 都走 www 了！

# ✅ 正确路由
if url.startswith("/api/"):
    base = EDITH_BASE
else:
    base = API_BASE
```

**教训**：路由规则要简洁明确，避免"黑名单"式逻辑。

---

### Bug #7: search_id 格式不一致

**症状**：搜索返回 0 条结果

**根因**：search_id 的 base36 编码字母表不一致

```python
# ❌ RedNote（小写 + UUID）
alphabet = "0123456789abcdefghijklmnopqrstuvwxyz"
n = int.from_bytes(uuid.uuid4().bytes[:8], "big")

# ✅ RedCrack（大写 + 时间戳）
alphabet = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ"
e = int(time.time() * 1000) << 64
t = int(random.uniform(0, 2147483646))
return base36encode(e + t)
```

**教训**：编码格式必须完全一致，包括字母表大小写。

---

### Bug #8: page_size 过小返回空结果

**症状**：`page_size=5` 时搜索返回空

**根因**：服务端对 `page_size` 有最小值要求（≥20）

**解决方案**：`page_size=max(count, 20)`

**教训**：API 参数的隐性约束需要通过测试发现。

---

### Bug #9: code_status 类型不匹配

**症状**：扫码后轮询无响应

**根因**：API 返回 `code_status: 0`（整数），代码比较的是字符串 `"1"`

```python
# ❌ 类型不匹配
code_status = status_data.get("data", {}).get("code_status", "")
if code_status == "1":  # 0 != "1"

# ✅ 转为字符串
code_status = str(status_data.get("data", {}).get("code_status", ""))
```

**教训**：Python 的 `==` 不做类型转换，整数 0 ≠ 字符串 "0"。

---

### Bug #10: 搜索结果 note_id 位置错误

**症状**：搜索结果 `note_id` 为空

**根因**：`note_id` 在 item 层级（`item["id"]`），不在 `note_card` 里

```json
{
  "id": "6a50d1d80000000017029c37",      // ← 在这里
  "xsec_token": "ABmLAax...",            // ← 也在这里
  "model_type": "note",
  "note_card": {
    "display_title": "...",
    "image_list": [...]
  }
}
```

**教训**：API 响应结构不一定是"嵌套最深"的，要检查每一层。

## 三、关键流程

### 3.1 登录流程

```
1. XHSClient.create()
   ├── 访问 edith 首页 → 获取 acw_tc
   ├── POST /api/sec/v1/scripting → websectiga + sec_poison_id
   ├── 生成完整浏览器指纹 (80+ 字段)
   ├── POST /api/sec/v1/shield/webprofile → gid
   └── POST /api/sns/web/v1/login/activate → visitor web_session

2. login_qrcode()
   ├── POST /api/sns/web/v1/login/qrcode/create → qr_id, code
   ├── 显示二维码 + 轮询 status
   └── POST /api/sns/web/v1/login/qrcode/status → web_session

3. save_cookies() → config/cookies.enc (JSON)
```

### 3.2 请求签名流程

```
每次请求前（RequestSigner.auth_flow）：
1. 更新 loadts = timestamp_ms
2. 更新指纹 (update_fingerprint)
3. 生成 6 个安全头：
   ├── x-s          ← HMAC 签名
   ├── x-s-common   ← RC4 加密 b1 指纹
   ├── x-t          ← 当前时间戳
   ├── x-b3-traceid ← 随机 hex
   ├── x-xray-traceid ← 时间戳 + 随机
   └── x-rap-param  ← 搜索/首页等需要
4. 移除 CDN 敏感头 (sec-ch-ua*, sec-fetch-*)
```

### 3.3 Cookie 体系

| Cookie | 来源 | 用途 |
|--------|------|------|
| `a1` | 本地生成 | 设备标识 |
| `webId` | 本地生成 | 浏览器 ID |
| `websectiga` | scripting API | 安全 token |
| `sec_poison_id` | scripting API | 安全 token |
| `gid` | webprofile API | 设备指纹 |
| `acw_tc` | edith 首页 | CDN 反爬 token |
| `web_session` | login/activate | 登录凭证 |
| `loadts` | 每次请求更新 | 时间戳 |
| `webBuild` | 固定值 | 版本号 |

## 四、CDN 架构

小红书使用多 CDN 架构，不同域名走不同 CDN：

| 域名 | CDN | 反爬策略 |
|------|-----|---------|
| `edith.xiaohongshu.com` | TencentEdgeOne + Akamai | TLS 指纹 + cookie + x-rap-param |
| `as.xiaohongshu.com` | nginx | 较宽松 |
| `www.xiaohongshu.com` | TencentEdgeOne | 中等 |

**关键发现**：
- httpx 的 TLS 指纹被 TencentEdgeOne 识别为非浏览器
- aiohttp 的 TLS 指纹可以绕过
- `sec-ch-ua*` 等浏览器特征头会加重检测

## 八、常见问题

### Q: 登录后搜索返回空结果？

**A**: 检查以下几点：
1. `page_size` 是否 ≥ 20
2. `acw_tc` cookie 是否存在（需要先访问 edith 首页）
3. `web_session` 是否有效（可能已过期，重新 `rednote login`）

### Q: 请求返回 400？

**A**: 可能原因：
1. 浏览器指纹不完整（缺少字段）
2. websectiga 解码错误（乱码）
3. CDN 拦截（TLS 指纹问题）

### Q: 请求返回 461？

**A**: 安全挑战，可能原因：
1. IP 风险（换网络或代理）
2. 需要扫码验证
3. 频率限制

### Q: 代理配置？

**A**: 编辑 `config/settings.yaml`：
```yaml
client:
  proxy: "http://127.0.0.1:7890"  # Clash
  # proxy: "socks5://127.0.0.1:1080"  # SOCKS5
  # proxy: "none"  # 直连
```

## 九、开发建议

1. **始终对照 RedCrack**：任何新功能先看 RedCrack 的实现，避免自行猜测
2. **CDN 敏感**：小红书 CDN 会检测 TLS 指纹、HTTP 头、cookie 完整性
3. **先跑通再优化**：用 aiohttp 跑通全流程，再考虑替换
4. **日志要详细**：请求/响应的 status、headers、body 都要记录
5. **测试要全面**：单元测试 + 集成测试 + 实际 API 测试
